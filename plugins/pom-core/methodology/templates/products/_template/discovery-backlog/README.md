# Discovery Backlog

Opportunities the Trio is **shaping** but has not yet committed to building. Each opportunity is one markdown file.

## What goes here

- Use cases routed from `../../../intake/use-case-backlog/` that this product owns
- Trio-originated opportunities not in the use case backlog yet (still get retrofitted into intake for visibility)

## What does NOT go here

- Items ready for a pod to pull → those move to [`../product-backlog/`](../product-backlog/)
- Unshaped wishlist items → those go back to intake or get killed

## Shape, then promote

The Trio's job in Discovery is to answer **four questions** before an item is promoted to the Product Backlog:

1. **Desirable** — do customers want it? (PD owns the answer)
2. **Viable** — does it serve the product's outcomes? (PM owns)
3. **Feasible** — can pods build it on the existing runway, or what runway investment is needed first? (TL owns)
4. **Ethical/compliant** — does it pass the standards in [`../../../enabling/`](../../../enabling/)?

An item that cannot answer all four is **not** ready to promote. See `the pom-core plugin's methodology/references/four-question-gate.md`.

## File format

`DISC-YYYY-MM-<short-slug>.md` — see [`DISC-template.md`](DISC-template.md). Each file references the originating intake entry, the current shaping state, and the four-question answers as they crystallize.
